# ASK-028 参考答案

## 现状

`/commit` 当前实现（`src/commands/slash/builtins/commit.ts`）：
- 跑 `git status --porcelain` + `git diff --stat HEAD` + `git rev-parse --abbrev-ref HEAD`
- 把工作区状态格式化展示
- **不跑 git commit**
- 提示用户自己 `git add -A && git commit -m "..."`

## 自动 commit 暂未做的子问题

### 1. 不可逆副作用
- 普通 commit：`git reset --soft HEAD~1` 可恢复
- 已 push 后：force-push 才能撤回，对协作仓库高风险
- 在 LLM 调用链里随手 commit，错误成本远高于其它工具

### 2. message 生成质量
- 好的 commit message 需要看 diff + 本会话上下文 + 推断"为什么改"
- 当前没有专门的 commit-message-generator skill
- 让通用 LLM 看 diff 写 message，风格漂移大、容易漏掉关键信息（破坏性改动 / TODO）

### 3. 审批入口缺位
- 自动 commit 必须走 pendingApproval 流程：先把"将要 commit 的文件 + 草稿 message"展示给用户
- 当前 PendingApproval 只对工具调用接好（write / bash 等），尚未支持"git operation approval"这种独立类别

### 4. scope 决策
- 是 commit 全部 (`git add -A`) / 仅 staged / 仅本会话 changedFiles？
- 跟 `/diff` 的 `changedFiles` 集合应该如何对齐？
- 这是产品语义问题，不是技术问题

## 为什么 read-only 是当前正确选择

- 零回归风险：只读，不可能弄坏用户仓库
- 立即可用：用户已经能用 /commit 看快照、自己 commit
- 不锁死未来：等四个子问题解决，可以无痛升级到 `/commit <message>` 真提交

## 反例

- "永远不能 auto commit" → 太绝对；只是当前不该
- "必须用 GUI" → 与本问题无关，REPL 也能做
