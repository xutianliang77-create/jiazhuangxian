# ASK-009 参考答案

## 分层设计

| 层 | 形式 | 放什么 |
|----|------|--------|
| 元信息 | SQLite (`~/.codeclaw/data.db`) | id / 时间戳 / 状态 / token_cost / 关联外键 |
| 正文 | JSONL (`sessions/{sid}/transcript.jsonl` / `observations.jsonl`) | 消息全文 / 工具 stdout / 大段 LLM 输出 |

## 好处

1. **查询成本**：按 session 统计 token / 按 trace 聚合 observation 这类操作在 db 里走索引就是毫秒级；如果正文也在 db 里，row 体积大、索引膨胀、PAGER_CACHE 命中率差。
2. **append 友好**：大正文顺序 append 到 jsonl 非常便宜，天然按时间排好，`tail -f` / `grep` / `cut` 等 Unix 工具直接可用。
3. **迁移 / 备份粒度**：只备 db 保元信息、只备 jsonl 保正文都是可行路径；`markBodyMissing`（ASK-004）就是这个解耦的应用。
4. **SQLite 体积控制**：生产期 db 停在几十 MB ~ 几百 MB，`VACUUM` / `schema 迁移`都能在秒级完成。

## 代价

- 两处真理：必须在 repo 层（ObservationRepo / L1MemoryRepo）保持双写一致
- 跨机迁移要记得搬 `sessions/` 目录，不能只搬 db

## 常见扣分 / 幻觉

- 说"SQLite 不支持 blob" → 错，支持；选 jsonl 是设计取舍不是技术限制
- 说"JSONL 是二进制" → 错，是每行一个 JSON 文本
- 只答"省空间"不提索引 / append → 粒度不够
