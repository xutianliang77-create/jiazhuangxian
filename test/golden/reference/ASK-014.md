# ASK-014 参考答案

## 为什么拆 data.db / audit.db

### 写入特征不同 → 锁竞争策略不同
- `data.db`：sessions / tasks / steps / observations 等业务表，频繁 read+write、有事务
- `audit.db`：纯 append-only 链式结构，每条 event 的写入很轻
- 同库时业务事务的 fsync 会阻塞审计写；分库后两类锁完全独立

### 备份策略可以差异化
| 库 | 是否可丢 | 备份频率 |
|----|----------|----------|
| data.db | 可丢，重建后重头跑（用户可接受） | 周备 |
| audit.db | **不可丢**（合规 / 取证） | 实时同步到外部存储 |

分库后只对 audit.db 做高代价的强备份，省成本。

### 权限隔离 / 防自伤
- 可以对 audit.db 做 `chmod 400` / 不同 owner，让"业务进程改不了审计"
- 安全审计场景里，第三方系统可以**只挂载 audit.db** 做独立校验，业务库的 schema / 内容对它不可见

### 取证可移植性
- 链式哈希需要纯净环境验证；audit.db 可以单独打包发给安全团队
- data.db 含用户隐私（消息正文 path 等），不一定能跨团队共享

## 代价

- 应用层多一份连接（已封装在 `src/storage/db.ts` 与 `audit.ts`）
- 不能跨库 JOIN：但 CodeClaw 的关联仅用 ULID（trace_id / event_id），全局唯一无需 JOIN

## 不该出现的论点

- "SQLite 不支持事务" → 错，支持
- "必须用 PostgreSQL" → 个人版数据规模 SQLite 完全够，迁 PG 是企业版的事
