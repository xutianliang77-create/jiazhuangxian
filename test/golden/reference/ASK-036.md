# ASK-036 参考答案

## 三层根因

### 前端层

| 假设 | 验证 |
|---|---|
| EventSource 默认监听 `message` 事件，但服务端用了具名 event `event: chunk` | F12 Network → EventSource → Messages tab 看 raw bytes 是否含 `event: <name>` 行；前端要 `es.addEventListener('chunk', ...)` |
| sessionId query 不对（未 encodeURIComponent 或串错） | 浏览器地址栏看实际请求 URL；服务端日志看 `/v1/web/stream?sessionId=...` 收到的 id |

### 网络层

| 假设 | 验证 |
|---|---|
| 反向代理（nginx / caddy / cloudflare）开了 buffering，SSE chunk 没立刻 flush | `curl -N -H "Authorization: Bearer ..." http://127.0.0.1:7180/v1/web/stream?sessionId=X` —— 直接到服务端能收，过代理收不到 = 代理 buffer 问题；nginx 要 `proxy_buffering off` |
| 中间设施（idle timeout）30s 切链，但前端 OPEN 状态没更新 | tcpdump / 服务端心跳计数：codeclaw 默认 20s 心跳；如果代理 idle 设成 15s 就会 idle 关链 |

### 服务端

| 假设 | 验证 |
|---|---|
| sessionId 还没在 store 里绑定 emitter（前端先开 stream 再 POST /messages 创建 session） | 先 POST /v1/web/sessions 拿 sessionId，再 GET /v1/web/stream；服务端 debug log 看 `store.getEmitter(sessionId)` 是否 hit |
| Bearer token 未带 / 错 → 401，但 EventSource 浏览器实现行为奇怪 | curl 不带 token 看 401；浏览器 EventSource 不能自定义 header，要么用代理添加，要么走 query token |
| 路径或方法 dispatch 错位（PATCH/DELETE 命中静态路由） | 服务端 dispatch log 看 url.pathname 实际命中哪个 handler |

## 最快诊断顺序

1. **`curl -N`** 直接到 7180 看 chunk 出不出 → 隔离前端 vs 服务端
2. 服务端 debug log 看 emitter 是否 attach、SSE write 调用次数
3. 浏览器 F12 Network → EventSource raw bytes

## 评分要点

- 至少答出 SSE event name 不匹配 / 代理 buffer / sessionId 绑定 / 心跳 超时其中 4 个
- 必须给出 curl 验证手段
- 不应推荐"重启电脑 / 重装浏览器"

## 常见扣分

- 推荐重启 / 重装：直接 fail
- 只说"加日志"不说看什么：-2
