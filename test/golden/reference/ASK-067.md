# ASK-067 参考答案

## /forget --all 的实际行为

| 范围 | 行为 |
|---|---|
| `data.db` | DELETE 跨表：dedup / cost / approvals / sessions metadata / providers cache |
| `audit.db` | **不删**：审计链 append-only。写一条 `forget` 事件（含被 forget 的范围 + 时间戳）作为 tombstone |
| `sessions/<id>/` 目录 | 实现而定：保留 / rename 为 `.deleted-<id>/` / 物理删（按当前版本检查） |

## 为什么 audit 不删

- hash 链断了就**永远断了**，再也不能 verify
- 即使用户"忘了"对话，审计要能回答"什么时候 forget 了什么"
- /forget 设计目标是"业务侧不可见"，不是"硬盘抹除"

## 验证

```bash
# 业务侧应已为空
sqlite3 .codeclaw/data.db "SELECT COUNT(*) FROM dedup; SELECT COUNT(*) FROM cost;"

# 审计应有 forget 事件
sqlite3 .codeclaw/audit.db "SELECT * FROM events WHERE kind='forget' ORDER BY ts DESC LIMIT 5;"

# 链未断
codeclaw forget verify-chain
```

## 如果真要"硬抹除"

应分两步：
1. `/forget --all` 业务清理
2. 在了解后果（链断、不可恢复）后用 `rm -rf .codeclaw` 全删 + `codeclaw setup` 重建

不要直接跳到第 2 步 —— 失去 forget 的审计记录。
