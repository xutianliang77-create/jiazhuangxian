# ASK-056 参考答案

## 关系图

```
POST /v1/web/sessions   ──→ store.createSession() ──┐
                                                     ├─→ {engine: QueryEngine, emitter: EventEmitter}
                                                     │     存进 sessionId → record map
GET  /v1/web/stream     ──→ store.getEmitter()  ────┘     订阅 emitter，把 chunk 写 SSE
POST /v1/web/messages   ──→ store.getEngine()    ────→    调 engine.handleInput(...)，engine 内部 emit chunk
```

## 创建时机

- emitter 在 `store.createSession()` 时创建（不是 stream 接入时）—— 这样允许"前端先 POST 拿 sessionId，再 GET stream，再 POST messages"任何顺序
- session 通过 `store.deleteSession()` 显式销毁；emitter 同步关闭

## 为什么解耦

| 场景 | 解耦后行为 |
|---|---|
| 前端断网重连 | engine 还在跑，新 SSE 接入读后续 chunk；不用重新触发 |
| 多个 tab 监听同 session | 多个订阅者各自收到 chunk |
| message 发送早于 stream 订阅 | emitter 缓存最近 chunk（如有 buffer 设计）/ engine 仍能跑完 |

不是 websocket（HTTP/SSE 单向流），不需要 express middleware（用 Node 内置 http）。
