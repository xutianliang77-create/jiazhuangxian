# ASK-068 参考答案

## 5 步定位

### 1. 检查 hash 一致性

```sql
SELECT id, prev_hash, payload_hash, ts
FROM events
WHERE id IN (... 断点前后 5 条 ...)
ORDER BY id;
```
- 内容字段被改：`SHA256(payload)` 与存储的 `payload_hash` 不一致
- prev_hash 损坏：`prev_hash` 不等于上一条的 `payload_hash`

### 2. timestamp 单调性

断点处时间是否回拨 / 巨大跳变 → 系统时钟乱了（断电 / NTP）。

### 3. 系统日志

```bash
journalctl --since='昨天 18:00' | grep -E 'kernel|fs|EXT4|btrfs'
dmesg -T | tail -200
ls -la .codeclaw/audit.db*
```
找 `IO error` / `fsck` / `panic` —— 定到具体硬件事件。

### 4. 比对备份

```bash
ls -lt .codeclaw/backups/  # 找最近 audit-*.db 备份（如有）
sqlite3 backup.db "SELECT id, payload_hash FROM events WHERE id BETWEEN x AND y"
diff <(...) <(...)
```

### 5. verify-chain 详细模式

```bash
codeclaw forget verify-chain --verbose
```
直接定位"第 N 条事件 hash 不匹配，期望 X 收到 Y"。

## 修复策略

| 判定 | 处理 |
|---|---|
| 自然损坏 + 有备份 | 从备份恢复 audit.db（保留破损版作 dump） |
| 自然损坏 + 无备份 | 从断点 fork 新链 + 写一条 `admin-reset` 事件记录原因；但**不要硬覆盖** |
| 真实篡改 | 保留 audit.db 作证物；冻结写入；查 ssh `last` / 同时段 sessions / 网络流量；走应急响应流程 |
| 时钟回拨 | 自然损坏分支；事后看 NTP 配置 |

## 不要做

- `rm audit.db` 重建：丢历史
- `codeclaw forget --skip-verify`：把"链断"当 feature
- 直接 `update events set payload_hash = ...` 修 hash：等于自己篡改证据
