# ASK-081 参考答案

## 优先级

| 工具 | 优先 | 原因 |
|---|---|---|
| **symbol (LSP find references)** | ⭐⭐⭐ | 类型驱动；区分函数 / 字符串 / 注释；跨文件追全 import 链 |
| grep / rg | ⭐⭐ | 快、跨语言、能看注释；但会误匹配同名 |
| read | ⭐ | 不是搜索工具，用于看具体文件 |

## 推荐流程

1. **symbol find-references** `validateManifest` —— 拿到所有 caller location
2. 如果 symbol 不可用（LSP 没装 / TS server 卡死）→ `rg -n '\bvalidateManifest\b'` fallback
3. 对每个 caller，read 看上下文（决定是否相关）

## grep 的陷阱

```ts
// 这条会被 grep 命中，但其实是字符串字面量，不是真调用
const docs = "see validateManifest for details";
```

LSP 不会把这种当 reference。

## read 的合理用途

- 看 `validateManifest` 自身实现
- 看具体某个 caller 的完整逻辑
- 都不是"找所有 caller" 这件事的主工具
