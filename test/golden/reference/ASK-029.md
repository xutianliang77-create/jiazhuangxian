# ASK-029 参考答案

## 三个工具的取舍

### `/replace` 单文件
- 一次替换一个文件里的"找/换"
- 30 个文件 × 30 次调用 + 30 次 ask（plan/default 模式下）= 太慢
- 适合"每个文件改的内容不同"的场景；这里所有文件改法一样，浪费

### `/bash` + `sed`
```bash
/bash "find src -name '*.ts' -exec sed -i 's|from \"../../old-utils\"|from \"@/utils\"|g' {} +"
```
- 一条命令搞定，最快
- **风险大**：
  - sed 默认 `-i` 直接改，无 dry-run，错了无回头
  - 路径里 `/`、`@` 容易让 sed 表达式翻车（要换分隔符为 `|` 或 `#`）
  - 不会跑测试，没法立即验证
  - macOS 的 sed `-i` 要 `-i ''` 加空字符串
- 适合"我已经做过 dry-run 完全确定"的精确替换

### `/orchestrate "rename old-utils import to @/utils"`
- planner 会规划：grep 出候选 → 批量 replace → 跑测试 → 出 diff 报告
- 每一步 medium+ 操作走 pendingApproval
- 失败 / 部分失败可以 reflector 决策
- 适合"批量 + 需要看 diff + 想稳"的场景

## 实战推荐路径

```
1. /grep 'from "../../old-utils"'        # 拉候选清单（cheap，read-only）
2. 看清单确认范围
3. /orchestrate "rename old-utils import to @/utils"  # 走完整管线
4. /diff                                  # 复核改动
5. 自己 git add + commit
```

## 不该做

- 手动 30 次 /read /write（违反"批量自动化"的精神）
- 必须开新分支（这是 git 习惯问题，不属于工具选择）
