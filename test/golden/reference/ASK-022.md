# ASK-022 参考答案

## 6 种 PermissionMode 对 medium 风险动作的处理

| 模式 | 行为 | 备注 |
|------|------|------|
| `default` | **询问用户**（写入 pendingApproval，等 /approve） | 平衡安全与便捷，多数交互场景 |
| `plan` | **询问用户** | 名义上"只产计划"，但 medium 仍需用户确认才能执行；与 default 在审批语义上等价，区别在于 LLM 默认走 read-only 思维 |
| `auto` | **自动通过** | low/medium 自动放行，仅 high 风险（如 rm -rf）才 ask |
| `acceptEdits` | **自动通过** | write/append/replace 等编辑类自动允许；bash 等仍 ask |
| `bypassPermissions` | **自动通过** | 所有风险等级自动放行，但仍写审计链 |
| `dontAsk` | **按缓存决策** | 之前用户已批/拒同类操作 → 复用决策；无缓存 → **拒绝** |

## low / high 行为外推

- low（如 /read /grep）：除 dontAsk 无缓存外，全模式自动通过
- high（如 /bash rm）：default / plan / auto / acceptEdits 都要 ask；只有 bypassPermissions 自动通过；dontAsk 无缓存仍拒

## 关键差异

- `acceptEdits` vs `auto`：前者只对 edit 类宽，bash 仍 ask；后者对 medium 一刀切宽
- `bypassPermissions` 和"完全没审计"是两回事：审计仍写，只是不打扰用户
- `dontAsk` 默认拒，避免新动作意外通过

## 不该出现的回答

- "全模式都直接通过" → 完全错
- "全模式都拒绝" → 完全错
- 漏列 mode 名 → 扣分
