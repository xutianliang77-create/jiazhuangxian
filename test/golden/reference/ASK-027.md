# ASK-027 参考答案

## 当前模式

每个 builtin 文件顶上声明它需要的最小子集：

```ts
interface StatusHolder {
  buildStatusReply(): string;
}

function isHolder(x: unknown): x is StatusHolder {
  return !!x && typeof (x as StatusHolder).buildStatusReply === "function";
}

handler(ctx) {
  if (!isHolder(ctx.queryEngine)) {
    return reply("status command unavailable: ...");
  }
  return reply(ctx.queryEngine.buildStatusReply());
}
```

不 `import { QueryEngine }` 类型本身。

## 动机

### 1. 避免循环依赖
- `src/agent/queryEngine.ts` 已经 `import { SlashRegistry, loadBuiltins } from "../commands/slash"`
- 如果 builtin 反向 `import type { QueryEngine }`，TS 类型层有循环（运行时 type-only 通常被擦，但仍然让 IDE / tsc 增量编译变慢且容易死循环）

### 2. 接口契约局部化
- 每个命令文件就是它对 host 的依赖契约
- `/status` 只用一个方法，文件顶部就一行声明，比"我用了完整 QueryEngine"更诚实
- code review 时一眼就能看出"这条命令会动什么"

### 3. 测试便利
- 单测不需要构造完整 QueryEngine（重）；给一个 plain object 实现同名方法即可
- 见 `test/unit/commands/slash/builtins.test.ts` 里大量这种最小 holder

### 4. 渐进迁移
- W2 期 QueryEngine 仍在重构，public API 不稳定
- duck-type 让 builtin 不需要跟随 QueryEngine 重构同步改

## 代价

- 编译期不会捕获"QueryEngine 把 buildStatusReply 重命名了但 builtin 没改"
- 重构 QueryEngine 公共方法时需要全局搜 builtin 文件确认
- 类型推断比直接 import 弱（IDE 跳转跳不到方法实现）

## 缓解措施

1. 每个 builtin 在 holder 缺方法时返回明确的 "command unavailable"，不崩溃
2. 测试覆盖两条路径：方法存在 → 委托；方法缺失 → graceful degrade
3. 关键重构时人工搜：`grep -rn "buildStatusReply" src/commands/slash/builtins/`

## 何时应反向，import 真实类型？

当：
- 命令固化、host 类型稳定（无再重构计划）
- 某 builtin 用到 QueryEngine 大量字段，duck-type 写起来累赘
- 性能敏感场景需要 inline / 优化

那时可以局部转成 `import type { QueryEngine }`。但 W2 阶段还早。
