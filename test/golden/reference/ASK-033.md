# ASK-033 参考答案

## 答题要点

`ProviderErrorClass` 在 `src/provider/chain.ts` 定义 5 种，每种处理策略如下：

| 错误类 | 触发 | 策略 |
|---|---|---|
| `transient` | 5xx / 429 / 408 / 425 / `ECONNRESET` / `ETIMEDOUT` / `ENOTFOUND` / `socket hang up` 等 | 同 provider 内 **retry**，默认 2 次，指数退避 `base * 2^(n-1)`，base 默认 500ms |
| `auth` | 401 / 403 | **不 retry**，立即 **fallback** 到下一个 provider |
| `content` | 413 / 422（请求负载本身有问题） | **不 retry 不 fallback**，直接抛错（修了 prompt 才能解决） |
| `abort` | 用户 Ctrl-C / `AbortSignal` 触发 | 直接抛出 AbortError，**不再尝试任何 provider** |
| `unknown` | 无法分类 | 兜底走 transient 路径（retry 一次后 fallback） |

## 关键约束

- **producedAny 守卫**：只要当前 attempt 已经向上游 yield 过任何 chunk（开始 streaming 后），即使后面失败也不允许 retry/fallback —— 否则会重复输出。chain 在第一个 token 抵达前才允许重试。
- attempts 数组完整记录每次尝试的 `{provider, attemptNo, ok, errorClass, errorMessage, durationMs}`，runner 用此打印失败链。

## 评分要点

- 5 种错误类至少答出 4 种（transient / auth / content / abort 必答）
- retry / fallback 概念必须出现
- 加分项：提到 producedAny / streaming 后不重试

## 常见扣分

- 把 401/403 归到 transient：-1（混淆）
- 没区分 retry vs fallback：-1
