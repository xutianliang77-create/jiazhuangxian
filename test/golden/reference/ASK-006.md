# ASK-006 参考答案

## 答题要点

`src/approvals/store.ts` 的 pending 列表查询使用 `ORDER BY ROWID ASC` 而非 `ORDER BY created_at ASC`，原因：

- `created_at` 用毫秒精度（Date.now()）
- 批量场景下同一毫秒可能有多条 pending 审批入库
- 同毫秒内按 `created_at` 排序 → 顺序由 SQLite 自选，**不稳定**
- `ROWID` 是 SQLite 内部单调递增（INTEGER PRIMARY KEY），严格反映"插入顺序"

## 改回 created_at 的 bug

- 用户看到的 pending 列表与实际插入顺序不符
- 用户以为在批准"较早的请求"时，可能批准了"较晚但同毫秒的"
- 审计回放里事件顺序与用户决策顺序不匹配，排查事故时困惑

## 复现办法

单测里连续两次 `recordPending(...)` 无 `await sleep(1)`，再 `loadPendingApprovals()`：
- `ORDER BY created_at` → 顺序可能颠倒
- `ORDER BY ROWID` → 稳定返回插入顺序

## 常见扣分 / 幻觉

- 说"ROWID 更快" → 不准确（索引已在 PK 上）
- 说"created_at 不可靠因为时钟回拨" → 不是这里的主要原因
- 说"ROWID 会重用已删除行" → 与排序顺序无关，跑偏
