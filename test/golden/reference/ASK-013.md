# ASK-013 参考答案

## 三类根因

### 1. 内容被人为篡改（最严重）
- 表现：链断点处某条 event 的 `hash` 与 prev_hash 不匹配，**该 event 之后的所有 hash 全部失配**（一断到底）
- 含义：有人/某代码改了 audit_events 行的 actor/action/decision 等关键字段
- 处置：作为安全事件 escalate；与备份 diff 找出被改字段

### 2. 顺序错乱 / 时钟问题
- 删除中间 event：链断在被删那条的下一条
- 插入伪造 event：连接处一定 hash 失配
- 时钟回拨：timestamp 非单调递增（链不一定断，但语义不对）
- 处置：同时核对 timestamp 单调性 + prev_hash 链；翻系统日志看 NTP 跳变

### 3. 自然损坏（可恢复）
- 磁盘掉电时部分写、文件系统 corruption、db 文件被替换为旧备份
- 表现：通常是连续多条 event 段断裂（不是单条）；hash 字段可能含非 hex 字符
- 处置：fsck / 恢复备份 / 重建链（接受历史断点，记下断点元数据）

## 怎么定位"篡改"vs"自然损坏"

| 信号 | 篡改 | 自然损坏 |
|------|------|----------|
| 断点附近 hash 字段格式 | 仍是合法 hex | 可能含乱码 / NUL |
| timestamp 单调性 | 通常仍单调 | 可能跳变 |
| 行数附近文件写入痕迹 | 可能正常 | 通常多个连续 page 受影响 |
| 系统日志在断点时间 | 无异常 | 有 kernel panic / disk error |
| 与最近 backup diff | 单字段差异 | 整段缺失或乱码 |

## 不该出现的回答

- "回滚区块链"——CodeClaw 不是区块链，没有共识层
- "关闭审计"——审计本来就是检测篡改的最后防线，关掉等于自废武功
