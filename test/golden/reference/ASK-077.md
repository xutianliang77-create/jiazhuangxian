# ASK-077 参考答案

## 设计

| 体积 | 落地形式 | db 字段 |
|---|---|---|
| ≤ 100 KB | 直接写入 `messages.jsonl` 一行 | `body_inline = 1` |
| > 100 KB | 独立文件 `sessions/<sid>/observations/<stepId>.jsonl` | `body_inline = 0` |

## 为什么不直接合并

| 风险 | 说明 |
|---|---|
| 单行过长 | jsonl 是行流；单行 5 MB 让任何 line-by-line reader 一次读爆内存 |
| 主流文件被拖累 | 90% observation 都是几 KB，但 1 个 5 MB 让整个文件解析慢 |
| 增量同步不友好 | rsync / git diff / 编辑器都按行操作，巨长行体验差 |

## 阈值选 100 KB 的理由

- 工具输出 99% 在这之下（read 行限制、grep 输出、ls 输出）
- 巨型工具结果（read 大文件、网络 fetch）才超 → 这些数据本来就该 lazy load
- 不会让"分两处"频繁触发

## 代价

- reader 必须看 `body_inline` 决定是否打开外置文件
- 备份要同时带 messages.jsonl + observations/ 子目录
- /forget 删 session 时记得清两边

## 不是

- 不是加密分离（两边都明文）
- 不是防止合并（合并也能通过 body_inline 字段重建）
