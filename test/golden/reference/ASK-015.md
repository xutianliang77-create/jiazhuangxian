# ASK-015 参考答案

## 为什么 reflecting 不合并到 executing

- **三段式分工**：CodeClaw 的编排是 Planner → Executor → **Reflector** → 决策
- Reflector 输入是 executor 产出的 observations + gaps，做"是否达成 / 是否要 replan"判断
- 合并后无法回答两类常见运维问题：
  - "工具跑得慢" vs "反思阶段慢"（前者优化工具，后者优化 reflector LLM 调用）
  - "卡在执行" vs "卡在决策"（用户 UI 上不一样，前者更难中断）
- 监控 P50 / P99 latency 也要分开统计，否则平均值失真

## 为什么 compacting 单列

- compaction（自动压缩历史）可以在 **任何 phase** 里被触发：
  - 主动：用户跑 `/compact`
  - 反应性：`reactiveCompactCount`（context 接近上限自动跑）
- 它和 executing / reflecting 的语义维度正交：不是"工作进度"而是"内务管理"
- 单列的好处：
  - `/status` 能告诉用户 "正在压缩历史，请稍等"，避免误以为卡死
  - 监控可以看到 compacting 占总时间比例（对评估 context 策略很关键）
- 如果藏进 executing，executing latency 分布会出现长尾，根因排查困难

## 一句话总结

phase 的粒度是"对外可观测度"决定的，不是按代码块物理位置分。
合并会让指标失真、用户感知错位、运维定位变难。
