# ASK-012 参考答案

## 排查 SQLite WAL 锁的步骤

1. **找占用进程**
   ```bash
   lsof ~/.codeclaw/data.db        # macOS / Linux 看哪个 PID 持有
   fuser  ~/.codeclaw/data.db      # Linux 替代
   ```
2. **查 WAL 状态**
   ```bash
   ls -l ~/.codeclaw/data.db-{wal,shm}
   sqlite3 ~/.codeclaw/data.db "PRAGMA wal_checkpoint(TRUNCATE);"
   ```
   如果 -wal 文件几十 MB 没回收，说明上次 codeclaw 没走 close，checkpoint 没跑。

## 最常根因

- 上一个 codeclaw 进程崩了 / 被强杀，没机会 `db.close()`
- iLink WeChat worker 子进程 hang 住，仍持有 db 连接
- SDK / gateway 模式的长连接在用户切走后没释放
- 多实例同时跑（用户开了两个 codeclaw 同时连同一个 ~/.codeclaw）

## 应急处理 vs 根因修复

**应急（数据安全前提下）**：
1. `kill <pid>` 终结占用进程（先确认 pid 非自己当前会话）
2. 确认无活跃连接后删 `*.db-shm` `*.db-wal` 残留
3. 重起 codeclaw，迁移会重新 checkpoint

**根因修复**：
- `process.on("exit"|"SIGINT"|"SIGTERM", () => db.close())` 全覆盖
- worker 心跳 watchdog：N 秒无心跳父进程 kill 子进程并清 lock
- /doctor 启动前自检：检测 stale wal，给用户 actionable 提示

## 不要做

- 不要"重装系统"或"卸载 SQLite"——这是文件级 lock，与系统无关
- 不要直接 `rm data.db`，会丢全部 session 元信息
