# ASK-076 参考答案

## 选 SQLite 的理由

| # | 维度 | 说明 |
|---|---|---|
| 1 | **单进程嵌入** | codeclaw 是 CLI 工具，没有 server。SQLite 与进程同生命周期，零部署 |
| 2 | **单文件** | data.db / audit.db 各一个文件；备份 = `cp`；迁移 = 复制文件 |
| 3 | **跨平台** | better-sqlite3 在 linux / macOS / Windows 都有 prebuilt 二进制 |
| 4 | **事务原子性** | 满足 codeclaw 写规模（每秒几条），ACID 完整支持 |
| 5 | **WAL 模式** | 读写混合不互相阻塞；codeclaw 启动时会切 WAL |
| 6 | **本地优先哲学** | codeclaw 是 personal-edition；不假设有云数据库 |

## PostgreSQL / MySQL 不适合的原因

它们是**网络数据库**：
- 需要 server 进程（增加部署复杂度）
- 需要网络/socket 通信（开销）
- 优势是高并发写、跨主机访问 —— 但 codeclaw 不需要

## SQLite 的弱点（不是选它的理由）

- 高并发写：所有写串行（WAL 缓解但不消除）
- 跨主机：不支持
- 分布式：无原生方案

codeclaw 都不需要这些，所以弱点不构成约束。
