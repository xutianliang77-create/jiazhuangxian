# ASK-042 参考答案

## 推荐顺序

| 步骤 | 工具 | 命令 | 目的 |
|---|---|---|---|
| 1 | grep | `rg -l '\bprocessOrder\b'` | 统计涉及文件数 |
| 2 | grep | `rg --json '\bprocessOrder\b'` 或 `rg -n '\bprocessOrder\b'` | 看每处具体行（区分用法） |
| 3 | LSP（如可用） | `Rename Symbol` | 类型驱动重构，最安全 |
| 4 | edit replace_all | 每个文件用足够上下文的 old_string | 精确替换 |
| 5 | bash | `npm run typecheck && npm test` | 验证 |

## 风险点

| 风险 | 例子 | 对策 |
|---|---|---|
| **子串误匹配** | `processOrderItem`、`processOrderTotal` | 用 `\b` 词边界（rg 默认是子串匹配，要 `-w` 或正则 `\b`） |
| **字符串字面量** | `console.log("processOrder failed")` | 替换前看每处上下文，决定改不改 |
| **SQL/JSON 字面量** | `"INSERT INTO logs ... 'processOrder' ..."` | 通常不改 |
| **跨语言重名** | TS 改了，Python/SQL/protobuf 还有 | rg 全语言扫描；如有跨语言契约要联动 |
| **注释不一致** | 有的注释改了，有的没改 | 决定一个原则：全改 / 全不改 |

## 反例（不推荐）

```bash
# 一刀切，会误伤 processOrderItem
sed -i 's/processOrder/handleOrder/g' **/*.ts

# 没词边界，没 typecheck，risky
find . -type f -exec sed -i 's/processOrder/handleOrder/g' {} \;
```

## 评分要点

- 必答：grep 先定位 → 词边界 → edit / LSP rename → typecheck 验证
- 必答：风险点至少 2 个（子串 / 字符串字面量）
- 不应推荐 `sed -i` 不带词边界一刀切

## 常见扣分

- 直接 `sed -i ... -g`：直接 fail
- 漏 typecheck 验证：-2
