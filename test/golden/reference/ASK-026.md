# ASK-026 参考答案

## 定位

vitest 默认发现规则会扫整仓 `*.test.ts`、`*.spec.ts`。
Golden FIX workspace 的 `test/router.test.ts` / `test/reducer.test.ts` 等被自动收进。

这些 workspace 是 Golden Set 故意 broken 的素材：
- FIX-01 router 故意 swap 了 key/value
- FIX-02 reducer 故意没 default case
- 由 `test/golden/runner/fix.ts` 隔离 venv / node_modules 后单独跑

进主回归就一定 fail —— 但这不是 bug，是配置缺失。

## 修复

根目录建 `vitest.config.ts`：

```ts
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    exclude: [
      "**/node_modules/**",
      "**/dist/**",
      "test/golden/fix/**",   // ← 关键
    ],
  },
});
```

重跑 `npm test`，应回到 baseline（FIX workspace 的测试不再被加载）。

## 为什么不直接删测试文件

那些测试是 Golden FIX runner 的 ground truth：
- runner 跑 `npm test` 在 workspace 内，先确认 verify_broken（必须有 fail）
- LLM 修完后，runner 跑 post_verify（必须 0 fail）
- 删了就没法判定是否修对

## 为什么 W2 早期没遇到

W1 收尾时 Golden 只有 ASK，没有 FIX workspace。引入 FIX 素材后才暴露。
现已通过 vitest.config.ts 排除并提交（commit `46e309c`）。

## 不该做

- 删测试文件
- 改 tsconfig（与测试发现无关）
- 给主测试加 `describe.skip()`（在每个 FIX 文件里都改一遍，不可持续）
