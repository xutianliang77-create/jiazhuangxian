# ASK-034 参考答案

## 可能根因（至少 4 类）

| 类别 | 描述 | 验证手段 |
|---|---|---|
| **provider 不返 usage** | 大多数本地 LLM（LM Studio、Ollama、llama.cpp）默认不返 `usage` 字段；OpenAI 流式必须 `stream_options: {include_usage: true}` 才返 | curl `/v1/chat/completions` 看末尾 chunk 是否含 `usage`；或开 codeclaw debug 日志看 chain 收到的最后一帧 |
| **cost 表写入失败** | `data.db` 连接异常 / 文件权限 / WAL lock | `sqlite3 .codeclaw/data.db "SELECT COUNT(*) FROM cost"`；看 codeclaw 启动 stderr 是否有 sqlite 报错 |
| **pricing 全为 0** | 该 provider 的 input/output 单价未配（lmstudio / ollama 默认 0） | 看 `~/.codeclaw/providers.json` 或 doctor 输出该 provider 的 pricing 字段 |
| **session 错位** | `/cost` 默认查当前 session；如果 dedup 命中复用了旧 sessionId，或多个 channel 混用，可能落到别的 session | `/cost --all`（如支持）或 `sqlite3 SELECT session_id, sum(amount) FROM cost GROUP BY session_id` |
| **单位渲染错** | 累计是 micro USD，但渲染当 USD 直接显示，导致 0.00001 → 显示 0 | 看 cost 表原始数值；调试 cost-renderer |

## 验证顺序（建议）

1. 先 `sqlite3 SELECT COUNT(*) FROM cost` —— 0 行 → 写入路径问题；非 0 → 渲染问题
2. 非 0 但金额都为 0 → pricing 配置问题
3. 0 行 + provider 是本地 LLM → 几乎一定是 usage 字段缺失
4. 0 行 + provider 是 OpenAI 真账号 → 检查 stream_options.include_usage

## 评分要点

- 至少 4 类根因
- 每类有可操作的验证手段
- 加分项：sqlite3 直查 / curl 看 usage 字段
- 绝不能建议"删 data.db" / "关审计" 这种破坏性操作

## 常见扣分

- 推荐删除 db / 关审计：直接 fail
- 只罗列"可能 bug" 不给验证手段：-2
