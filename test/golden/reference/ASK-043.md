# ASK-043 参考答案

## 前 5 个动作（高信号低成本递进）

| # | 命令 | 识别什么 |
|---|---|---|
| 1 | `cat package.json package-lock.json && cat requirements.txt` | 依赖清单；可比对已知 CVE 库（lodash <4.17.21 / log4j 等价物 / mass-assignment 库 / pickle 反序列化库等） |
| 2 | `rg -i 'api[_-]?key\|password\|secret\|token\|bearer' --type-add 'all:*' -tall \| head -50` | 硬编码密钥；常见误提交格式 `OPENAI_API_KEY="sk-..."` |
| 3 | `ls -a && cat .env* 2>/dev/null; cat .gitignore` | 是否真把密钥提交了；.env 是否在 .gitignore |
| 4 | `rg -n 'eval\(\|exec\(\|child_process\|os\.system\|subprocess' src/` | 命令/代码注入面（动态 eval、shell exec） |
| 5 | `rg -n 'execute\(.*\$\|\.raw\(\|f"SELECT.*\{' --type py --type ts` | SQL 注入（字符串拼接 / f-string / .raw） |

## 后续可加（成本递增）

6. `rg -n 'CORS\|Access-Control' --type js --type ts` — CORS 配置
7. `rg -n 'http://' --type ts --type js` — 明文 HTTP（应改 HTTPS）
8. `git log --all --full-history --diff-filter=D -- '*.env' '*.pem' '*.key'` — 历史 commit 中曾提交又删除的密钥痕迹
9. `rg -n 'jwt\.decode\(\|verify.*false' --type ts` — JWT 校验缺失
10. 跑 `npm audit --omit=dev` / `pip-audit` —— 自动化已知 CVE 扫

## 为什么这个顺序

- **依赖优先**：1 条命令能覆盖几十个已知 CVE，性价比最高
- **密钥次之**：直接危害（信息泄露），rg 几秒出结果
- **代码注入第三**：需要稍微读上下文判断
- **协议/CORS 第四**：通常 false positive 高，要逐个看

## 反例（高成本低信号）

- "静态扫描所有代码"：1 万文件 grep 无目标 → 输出过载
- "跑全量测试"：测试通过 ≠ 没安全洞
- "随机 read 几个文件看看"：浪费工具配额

## 评分要点

- 必须给出 5 条具体命令（不是抽象描述）
- 必须覆盖：依赖 / 密钥 / 代码注入 / SQL 注入 / .env 至少 4 类
- 必须按"高信号低成本"递进
- 不应一上来全量扫描

## 常见扣分

- 没给具体命令，只描述 "看一下密钥"：-3
- 5 条命令重复（如全是 rg）：-2
- 推荐 "随机抽样":  -2
