# ASK-086 参考答案

## 6 步定位

### 1. 进程拓扑

```bash
ps -ef | grep python | grep -v grep
ps -L -p <PID> | wc -l   # 线程数
ls /proc/<PID>/task/     # 子线程
```
判断：fork 出几十个 worker → 多半是 DataLoader `num_workers > 0` 卡 fork。

### 2. GPU 状态

```bash
nvidia-smi -i 0 -q -d UTILIZATION,MEMORY,PROCESSES
```
CPU 100% + GPU 0% + VRAM 已分配 → 数据 pipeline 卡在前向传播之前。

### 3. Python stack 抓取

```bash
pip install py-spy   # 已装则跳
py-spy dump --pid <PID>          # 一次快照
py-spy top --pid <PID>           # 持续 top
```
判断：
- 停在 `multiprocessing/queues.py:get` → DataLoader worker 卡 IPC
- 停在 `torch/cuda/...` → cuda 同步问题
- 在 `numpy/random` 死循环 → 数据增强逻辑 bug

### 4. syscall 层

```bash
strace -p <PID> -f -e trace=read,openat,futex 2>&1 | head -200
```
判断：
- 狂 `read` 同一个大文件 → IO 瓶颈（建议 num_workers + pin_memory）
- 卡 `futex` → 锁争用

### 5. 看代码 / log 配置

```bash
grep -nE 'num_workers|pin_memory|prefetch|print|logger' train.py
```
确认：
- `num_workers > 0` 但没设 `persistent_workers=True` → fork 重启代价高
- print / logger 没 flush → 输出被 buffer，看似"无输出"实则有

### 6. 干预性复现

```bash
# 强制无 buffer
PYTHONUNBUFFERED=1 python train.py --debug --epochs=1 --batch-size=4

# 临时禁 worker
python train.py --num-workers=0
```
看是否仍卡 → 隔离是 worker 还是主流程问题。

## 不要做

- `kill -9 <PID>`：直接杀，丢已经跑了 6 小时的训练结果（如果有 checkpoint 还好；没有就白干）
  - 优先 `SIGTERM` 让 PyTorch 保存 checkpoint
- 重启机器：损失更大 + 不知道根因下次还会再卡
- 改完代码盲跑：先小数据集复现，再上全量

## 常见根因（按概率高→低）

1. DataLoader `num_workers` 与 `pin_memory` 配置不兼容当前数据增强（fork 后 numpy random state 同步）
2. 输出 buffer 未 flush → 看似无输出实际在跑
3. 数据 IO 瓶颈（大量小文件读 + 慢盘）
4. CUDA driver / pytorch 版本不兼容卡 cuda init
5. 主进程死锁（多个 backward / optim step 互锁）
