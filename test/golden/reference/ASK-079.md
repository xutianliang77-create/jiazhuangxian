# ASK-079 参考答案

## 三者对比

| 维度 | better-sqlite3 | sqlite3 | sql.js |
|---|---|---|---|
| 实现 | 原生 C++ binding（synchronous） | 原生 C binding（callback async） | 纯 JS / WASM |
| API | `db.prepare(...).run(...)` 同步 | `db.run(sql, cb)` 异步 | 同步（内存） |
| 落盘 | 直接读写文件 | 直接读写文件 | 内存；export() 拿 Uint8Array 自己写盘 |
| 性能写入 | ~100k+ tps | ~50k tps | ~1k tps |
| 场景 | 嵌入 CLI / Electron / 服务端 | 老式 Node app | 浏览器 / 沙箱 |

## 为什么选 better-sqlite3

| 理由 | 说明 |
|---|---|
| **同步 API 自然** | codeclaw 是 CLI 顺序流；不用每次 `await db.run()` 嵌套 |
| **单进程零开销** | 无 worker / thread pool 调度；启动即用 |
| **性能超 codeclaw 需求** | 100k tps，codeclaw 写规模几条/秒，永远不会成瓶颈 |
| **prebuilt 覆盖** | linux/macOS/Windows × x64/arm64 都有 prebuild，无需用户装编译工具链 |

## 为什么不用 sql.js

- 纯内存：codeclaw 要持久化 sessions / audit，sql.js 落盘要自己 export 全 db，写 amplification 严重
- 性能慢百倍
- 适合的是浏览器侧的轻量查询，不是 codeclaw 这种服务端写密集场景

## 为什么不用 sqlite3 (老 npm 包)

- callback 风格强行异步，让简单流程变嵌套
- codeclaw 不需要的 worker 模型反而增加复杂度
- 维护活跃度低于 better-sqlite3
