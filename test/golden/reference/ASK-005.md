# ASK-005 参考答案

## 答题要点

普通 ULID 结构：`<48 位毫秒时间戳>` + `<80 位随机>`。
问题：同一毫秒内两次 `ulid()` 随机部分独立，**大小关系不确定**。

`monotonicFactory()` 返回一个记住上一次结果的生成器：
- 若当前毫秒 > 上一次毫秒 → 正常生成
- 若当前毫秒 == 上一次毫秒 → 把上一次的随机部分 **+1**（大整数自增），确保当前 id > 上一次 id

## 同一毫秒两次调用的关系

严格单调递增：`id_2 > id_1`（字典序），且仅比 `id_1` 大"一步"。

## 为什么 CodeClaw 需要这个

- `ORDER BY id` 直接等价于"按生成顺序"；无需额外 seq 列
- 审计事件、observation、step 都需要可排序
- 避免同一毫秒并发写入导致历史顺序错乱

## 常见扣分 / 幻觉

- 说"monotonicFactory 实现原子锁" → 不准确；它是用户态单实例状态，不跨进程
- 说"支持时钟回拨" → 不支持；时钟回拨会导致抛错或等下一毫秒
- 混淆 UUID v4 / v7 概念 → 扣分
