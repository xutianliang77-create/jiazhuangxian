# ASK-053 参考答案

## 根因：索引偏移

`scanForSecrets` 返回所有 finding 的 `[startIndex, endIndex]`。原文：
```
prefix sk-XXXXXXXXX middle ghp_YYYYYYY suffix
                 ^idx=10                ^idx=30
```
替换内容是 `[REDACTED:openai-key]`（长度不同于原 secret）。

**正序替换**：
- 第 1 处 idx=10 替换为 `[REDACTED:openai-key]`，假设新长度比原 secret 多 5
- 第 2 处 idx=30 在原文中是对的，但**新文本里 ghp_ 实际位置已移到 idx=35**
- 用旧 idx=30 splice → 截到 `[REDACTED:openai-` 的中间，破坏前一处替换 + 没正确替换 ghp

**倒序替换**：
- 先处理 idx=30：splice ghp_… → 替换。前面 idx=10 区域**完全不动**，索引不变
- 再处理 idx=10：splice sk-… → 替换。后面已经替换好的 `[REDACTED:openai-key]` 在新位置，但用不到旧 idx

## 通用原则

只要替换长度 ≠ 原长度，对同一字符串多处 splice 都必须 desc by index 处理。

与性能 / 多线程无关 —— 是 immutable string slice + 多次 splice 的固有顺序约束。
