# ASK-035 参考答案

## 三层根因假设

### 1. 代码层

`src/storage/migrate.ts` 中：
```ts
const dir = path.join(__dirname, "migrations", kind);  // kind = "data" | "audit"
```
- tsx 直跑（dev 模式）时 `__dirname` ≈ `src/storage/`，读 `src/storage/migrations/data/`
- 这层一般没问题，**除非 src 下子目录被误删**

**验证**：`ls src/storage/migrations/{data,audit}` 看 SQL 文件是否在
**修复**：从 git 恢复

### 2. 构建产物层（最常见根因）

打包后 `dist/storage/migrate.js` 期望读 `dist/storage/migrations/{data,audit}/NNN_*.sql`，但：
- `tsc` / `esbuild` 默认**不拷贝非 .ts 文件**
- `scripts/build.mjs` 必须显式 `cp -r src/storage/migrations dist/storage/migrations`
- 如果开发者裸跑 `tsc` 或 build.mjs cp 步骤被注释，dist 下就没 SQL

**验证**：
```bash
ls dist/storage/migrations/data/
ls dist/storage/migrations/audit/
```
任一空就是这个问题。

**修复**：检查 `scripts/build.mjs` 的 cp 步骤；重跑 `npm run build`。

### 3. 文件系统层

- 文件名不匹配 `^(\d+)_(.+)\.sql$`（如多了空格、缺前缀数字）→ migrate.ts 会 throw `bad migration filename`
- `migrations/` 被 `.gitignore` 误忽略（看 `.gitignore`，不应有 `migrations/` 模式）
- 软链丢失（CI/Docker 镜像构建中 -L vs -P 行为差异）

**验证**：`find src/storage/migrations -type f -name '*.sql'`
**修复**：rename 文件 / 改 .gitignore / 重新生成软链

## 不应做的"修复"

- **跳过 migration**：会让 schema 与代码版本不一致，后续 INSERT/SELECT 全报字段不存在
- **改 migrate.ts 容错忽略缺失**：同上，把"启动失败"换成"运行时随机崩"
- **手动 sqlite3 灌 SQL**：丢失版本号，下次启动还会重复执行

## 评分要点

- 必须答出"build/cp 漏拷"这条最高频根因
- 至少给出 2 层假设 + 验证命令
- 不应推荐"跳过 migration"

## 常见扣分

- 只答"重装"：-3
- 推荐跳过 migration：直接 fail
