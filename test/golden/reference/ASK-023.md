# ASK-023 参考答案

## 三步定位

1. **环境检查**：`which zsh`
   - 没装 → 环境问题（Ubuntu/Debian 默认只有 bash）
   - 装了 → 看测试是否硬编码 `shell: "zsh"` 而非 `process.env.SHELL`

2. **回归基线确认**：`git stash` 暂存所有改动 → 重跑同一条
   - 仍 fail → 是基线问题（不是本次改动引入），属于 known-issue
   - pass → 是新引入的回归，重点看 stash 里改了什么

3. **代码定位**：`grep -rn "zsh" test/` 找硬编码

## 应急方案

```bash
# 跳过这一条继续跑回归
npx vitest run --exclude test/local-tools.test.ts

# 或本地装 zsh
sudo apt install zsh   # Linux
brew install zsh       # macOS
```

## 根因修复

两条路：

```ts
// 路 A：测试侧加 skip 守卫
const hasZsh = (() => {
  try { execFileSync("which", ["zsh"], { stdio: "ignore" }); return true; }
  catch { return false; }
})();
(hasZsh ? it : it.skip)("runs low-risk bash commands", ...);

// 路 B：源头改成尊重用户 shell
spawn(process.env.SHELL ?? "bash", ["-lc", "pwd"], ...);
```

路 B 更彻底，但要确认产品逻辑是否依赖 zsh（看起来不依赖）。

## 不该做

- 卸载 npm（无关）
- 关掉所有测试（掩耳盗铃）
- 修改 package.json test 脚本永久排除（让回归门禁缩水）
